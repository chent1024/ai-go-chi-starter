# 通用工程约束

这个仓库不依赖 prompt 自觉执行规则。所有改动默认必须通过 `make verify`；涉及并发、存储、队列、runtime wiring、契约或 migration 的改动，必须通过 `make verify-strict`；如果仓库定义了 `make verify-perf`，性能相关改动必须额外通过它。

## 1. 强制验证

- 普通改动完成后必须执行 `make verify`
- 涉及并发、存储、队列、runtime wiring、契约或 migration 的改动，必须执行 `make verify-strict`
- 如果仓库定义了 `make verify-perf`，性能相关改动必须额外执行它
- 不允许把“代码已改但未验证”视为完成态
- 不允许用局部自定义命令替代仓库统一验证入口，除非任务明确要求且最终仍补跑对应的 `make verify`、`make verify-strict`、`make verify-perf`
- 用户请求“提交”或“推送”时，必须先完成对应验证，再执行 `git add`、`git commit`、`git push`
- `make verify`、`make verify-strict` 或仓库定义的 `make verify-perf` 未通过前，不得将提交、推送、已完成或“可稍后补验证”视为可接受状态
- 除非用户明确要求，不得建议或使用 `--no-verify`、跳过 hooks 或其他绕过仓库校验的做法

## 2. 约束入口

- `.orch/rules/` 是当前仓库的运行态约束目录；约束工具通过已安装的 `rules` / `rules-kit` 命令消费
- 调整 target 结构或受管默认约束时，优先修改 `.orch/rules/manifest.json`；仓库私有的路径或 env 覆盖项，优先修改 `.orch/rules/<target>/local.env`；只有在明确调整受管默认值时才直接修改 `.orch/rules/<target>/rules.env`
- 不要手工改 `Makefile.rules`；需要刷新生成结果时，优先执行 `rules install . --manifest .orch/rules/manifest.json --yes`；只有在迁移旧版 `AGENTS.md` 或需要显式覆盖受管 base 文件时才使用 `--force --yes`
- 除纯文档和小型说明性改动外，开始实现前默认先执行 preflight：优先 `rules preflight .`；若当前环境找不到 `rules`，则执行 `bash .orch/rules-kit/preflight.sh .`
- 如果改动命中仓库配置的 OpenSpec 受控路径，且 preflight 要求先创建 change，则必须先创建 OpenSpec change 再继续实现
- `CHANGE_SCOPE_ACTIVE_CHANGE` 仅用于多个 active OpenSpec changes 并存时的人工歧义消解，不是常规必填项
- 运行态 target 配置位于 `.orch/rules/<target>/`；不要假设默认 target 一定叫 `service`
- 不要把 `.orch/rules/` 当模板目录使用
- 根级 `AGENTS.md` 属于约束入口；不要随意删除或绕过
- 新增 env key 时，优先复用仓库现有配置读取入口与调用形态；避免引入仓库配置扫描规则无法识别的新 helper 或分散读取方式
- 新增 env key 时，默认同步 `.env.example` 以及仓库内已有的本地开发 / 部署示例 env 文件

## 3. 变更边界

- 修改前先阅读现有实现、测试和调用链
- 默认只做完成当前任务所必需的最小改动
- 不顺手做无关清理、批量重命名、目录搬迁、全仓格式化或依赖升级
- 不覆盖、不回滚用户已有的未提交改动；若与当前任务直接冲突，先说明冲突点
- 优先复用仓库现有模式、结构、技术选择、装配方式和测试风格，不因为个人偏好替换框架或重写分层

## 4. 分层与实现

- 传输层只负责协议转换，不直接依赖具体持久化实现
- 业务层优先依赖接口，不依赖传输层或具体存储实现
- 具体装配只放在 wiring / composition 层
- 不要把具体实现依赖跨层带入错误的位置
- 新增或修改代码时，默认保持函数和文件短小可拆分；避免把复杂逻辑堆进超长函数
- 单个生产函数原则上不超过 80 行；单行原则上不超过 120 列，除非 URL、生成内容或无法拆分的字面量
- 优先标准库和仓库已有依赖；新增、删除或升级第三方依赖时，必须说明为什么现有栈不足

## 5. 契约与配置同步

- 修改对外接口契约、配置加载、鉴权行为、状态流转、持久化行为或并发行为时，默认必须同步相关测试、文档和契约定义
- 修改 API surface 目录下的文件时，先评估是否需要同步契约文件；不要等严格校验失败后再补 contract
- 即使本次只改测试，也要预判仓库的 contract-sync / doc-sync 是否会把该目录视为 API surface 变化
- 新增或调整配置项时，除了代码，还要同步 `.env.example`、相关配置文档，并在交付说明里列出新增 key、默认值和影响到的 runtime
- 不允许把“文档、契约、配置以后再补”视为完成态
- 是否完成以仓库内可执行校验为准，而不是以主观判断为准
- generated 文件只能通过既有生成入口更新，禁止直接手改生成结果

## 6. 测试与安全

- 新增或修改测试时，默认不得访问真实外网，不依赖开发者本机私有状态或手工环境
- 涉及接口契约、状态机、存储、鉴权、并发、配置路径的改动，通常必须补齐对应测试
- 禁止在代码、日志、fixture、快照、示例配置、文档中写入真实 secret、token、key 或敏感数据
- 提交前检查 `git status --short`，确认 `tmp/`、`.runtime/`、本地对象存储目录等运行产物没有进入暂存区
- 如果同步了本地示例文件但该文件未受 git 跟踪，最终说明里要显式写明“已更新本地文件，但未进入版本控制”

## 7. 交付要求

- 如果验证失败，先说明失败来自代码、契约/文档未同步，还是仓库规则触发；在未定位失败来源前，不要声称任务已完成
- 交付说明必须简洁、可核对、可追责；不要只写“已完成”或泛泛描述
- 每次交付至少说明：
  1. 改了什么
  2. 同步了哪些契约 / 配置 / 文档；如无，写“无”
  3. 实际执行了哪些验证；如未执行，写原因
  4. 还剩哪些风险、假设或缺口；如无，写“无”
- 如果改动涉及接口契约、数据结构、状态机、存储、鉴权、并发、默认命令行为、兼容策略或规则生成结果，还必须额外说明：
  1. 是否存在 breaking change
  2. 是否需要迁移
  3. 回滚方式、兼容策略或未覆盖场景
